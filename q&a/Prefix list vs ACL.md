为了让你彻底掌握这两者的区别，我们来看 10 个典型的路由过滤场景。

**核心核心区别：** ACL 只能匹配**地址位**（使用通配符掩码 Wildcard Mask），而 Prefix List 可以同时约束**地址位**和**掩码长度（Prefix Length）**。

------

### 场景 1：精确匹配一个子网

- **目标：** 只允许 `192.168.1.0/24` 进入路由表。
- **ACL:** `access-list 1 permit 192.168.1.0 0.0.0.255`
  - *风险：* 它也会匹配到 `192.168.1.0/25` 或 `192.168.1.0/30`，因为它只看前 24 位是否匹配。
- **Prefix List:** `ip prefix-list EX1 permit 192.168.1.0/24`
  - *优势：* 只有网络号是 `192.168.1.0` 且掩码正好是 `24` 位的路由才通过。

### 场景 2：匹配一个大网段及其所有子网

- **目标：** 允许 `10.0.0.0/8` 以及它下面所有的子网（无论掩码是多少）。
- **ACL:** `access-list 1 permit 10.0.0.0 0.255.255.255`
- **Prefix List:** `ip prefix-list EX2 permit 10.0.0.0/8 le 32`

### 场景 3：匹配所有的主机路由 (Host Routes)

- **目标：** 抓取所有的 `/32` 路由。
- **ACL:** **无法实现**（ACL 无法识别路由的掩码长度）。
- **Prefix List:** `ip prefix-list EX3 permit 0.0.0.0/0 ge 32`
  - *解释：* 网段任意，但掩码必须 $\ge 32$。

### 场景 4：匹配所有 A 类私有地址的“标准”子网

- **目标：** 匹配 `10.x.x.x` 网段中掩码长度恰好为 `/24` 的路由。
- **ACL:** `access-list 1 permit 10.0.0.0 0.255.255.255` (同样无法限制掩码长度)
- **Prefix List:** `ip prefix-list EX4 permit 10.0.0.0/8 ge 24 le 24`

### 场景 5：放行默认路由 (Default Route)

- **目标：** 只放行 `0.0.0.0/0`。
- **ACL:** `access-list 1 permit 0.0.0.0`
- **Prefix List:** `ip prefix-list EX5 permit 0.0.0.0/0`

### 场景 6：匹配特定的超网 (Supernet)

- **目标：** 匹配 `172.16.0.0` 到 `172.31.255.255` 之间掩码在 `/12` 到 `/16` 之间的路由。
- **ACL:** `access-list 1 permit 172.16.0.0 0.15.255.255` (依然无法锁定掩码范围)
- **Prefix List:** `ip prefix-list EX6 permit 172.16.0.0/12 le 16`

### 场景 7：抓取所有的“互联地址”段

- **目标：** 假设互联地址统一使用 `/30` 掩码。
- **ACL:** **无法实现**。
- **Prefix List:** `ip prefix-list EX7 permit 0.0.0.0/0 ge 30 le 30`

### 场景 8：在 BGP 中过滤特定长度的路由条目

- **目标：** 拒绝所有掩码长度大于 `/24` 的路由（防止路由表过度碎片化）。
- **ACL:** **无法实现**。
- **Prefix List:** `ip prefix-list EX8 deny 0.0.0.0/0 ge 25` / `ip prefix-list EX8 permit 0.0.0.0/0 le 32`

### 场景 9：精确匹配 C 类地址空间

- **目标：** 匹配所有 `192.0.0.0` 到 `223.255.255.255` 范围内的路由。
- **ACL:** `access-list 1 permit 192.0.0.0 31.255.255.255`
- **Prefix List:** `ip prefix-list EX9 permit 192.0.0.0/3 le 32`

### 场景 10：放行所有路由 (Permit Any)

- **目标：** 类似 ACL 最后的 `permit any`。
- **ACL:** `access-list 1 permit any`
- **Prefix List:** `ip prefix-list EX10 permit 0.0.0.0/0 le 32`

------

### 总结对照表

| **需求**         | **ACL 实现**              | **Prefix List 实现**     | **评价**             |
| ---------------- | ------------------------- | ------------------------ | -------------------- |
| **匹配地址位**   | ✅ 支持 (Wildcard)         | ✅ 支持 (Prefix/Len)      | 平手                 |
| **匹配掩码长度** | ❌ **不支持**              | ✅ **原生支持 (ge/le)**   | **Prefix List 完胜** |
| **处理大路由表** | 🐢 线性查找，慢            | ⚡ Trie 树查找，快        | **Prefix List 完胜** |
| **应用场景**     | 接口流量过滤 (Data Plane) | 路由策略 (Control Plane) | 分工明确             |

**这就是为什么在 CCNP 考试中，只要涉及到 OSPF 重新分配（Redistribution）或 BGP 路由过滤，标准答案几乎总是 Prefix List。**

既然你现在能分清这两者了，你想看一看在 `route-map` 里如何结合 `prefix-list` 做更复杂的 **Policy-Based Routing (PBR)** 吗？

这个问题问得非常到位！既然 Prefix List 在路由过滤上几乎是碾压 ACL 的存在，为什么 ACL 还没有被淘汰？

核心原因在于：**它们服务的“平面”不同。**

简单来说：**ACL 是用来管“包”（Traffic）的，而 Prefix List 是用来管“路”（Routes）的。**

------

## 1. 职责分工：数据平面 vs 控制平面

### ACL：流量的“安检员”（Data Plane）

ACL 能够深入查看数据包的特征，而不不仅仅是 IP。

- **除了 IP，还能看端口：** ACL 可以区分 HTTP (80)、HTTPS (443)、SSH (22)。Prefix List 只能看网段，它根本不知道这个包是网页还是邮件。
- **除了 IP，还能看协议：** ACL 可以区分 TCP、UDP、ICMP（Ping）。
- **应用场景：** 防火墙过滤、QoS 流量限速、NAT 地址转换。如果你想禁止某个 IP 访问你的服务器，必须用 ACL。

### Prefix List：地图的“修正带”（Control Plane）

Prefix List 只存在于路由协议的更新过程中。

- **它不看数据包：** 当一个数据包经过路由器时，Prefix List 根本不起作用。它只在路由器互相交换“地图”（路由条目）时起作用。
- **应用场景：** BGP 路由过滤、OSPF 重分布过滤。它决定的是“我的路由表里应不应该有这条路”。

------

## 2. 为什么 ACL 无法取代 Prefix List？（反之亦然）

我们可以用一张表看清它们的“死穴”：

| **功能**                      | **ACL (Access Control List)**  | **Prefix List**      |
| ----------------------------- | ------------------------------ | -------------------- |
| **能匹配 L4 端口吗？**        | ✅ **可以** (抓取 80 端口流量)  | ❌ **不行**           |
| **能匹配协议类型吗？**        | ✅ **可以** (抓取 OSPF/ICMP 包) | ❌ **不行**           |
| **能区分 10/8 和 10/24 吗？** | ❌ **很难** (不检查掩码长度)    | ✅ **极佳**           |
| **性能瓶颈**                  | 处理上万条路由更新时会卡顿     | 处理海量路由依然丝滑 |

------

## 3. 形象的比喻：安检 vs 导航

- **ACL 就像机场的安检：**

  它盯着每一个**人**（数据包）。它会检查你带了什么（端口/协议），只要你经过大门（接口），它就扫描你。

- **Prefix List 就像高德地图的后台：**

  它盯着**路**（路由信息）。它决定在地图上是否显示“某条高速公路”（网段）。如果它把路删了，你（数据包）根本就找不到去那里的导航，连安检门都碰不到。

------

## 4. 你的 CCNP 备考金律

在考试中，你只需要看题目要你干什么：

1. **题目说：** “阻止主机 A 访问服务器 B 的 Telnet 服务。” —— **选 ACL**。
2. **题目说：** “在重分布时，只允许 10.1.1.0/24 路由进入 OSPF。” —— **选 Prefix List**。
3. **题目说：** “定义 NAT 的感兴趣流。” —— **选 ACL**（因为 NAT 匹配的是包）。
4. **题目说：** “在 BGP 中防止收到过细的碎片路由。” —— **选 Prefix List**。

------

## 总结

**ACL 是“广度”胜出**，什么都能管（IP, Port, Protocol），但不够“深度”（不看掩码）。

**Prefix List 是“深度”胜出**，专精于路由掩码和性能，但“广度”不行（只能看 IP 路由）。

这是 CCNP 考试和实际工程中最容易让人“CPU 烧毁”的地方。两者的计算逻辑完全不同：**通配符掩码（Wildcard Mask）是位运算的“反向匹配”，而 Prefix List 是长度的“区间锁定”。**

------

## 1. 通配符掩码：灵活的“滤网” (Wildcard Mask)

在 ACL 中使用。它的逻辑很简单：**0 必须匹配，1 代表无所谓（Don't Care）。**

### 易混淆点：它不一定是连续的

虽然我们常用 `0.0.0.255`（匹配前 24 位），但通配符掩码其实可以写出“奇葩”逻辑：

- **例子：** `10.1.1.0 0.0.0.254`
- **计算：** 最后一位是 `254`（即 `11111110`）。因为最后一位是 `0`，意味着最后一位 IP 必须匹配。
- **结果：** 这个 ACL 只能匹配到 **10.1.1.0, 10.1.1.2, 10.1.1.4...** 等所有的**偶数**地址。

------

## 2. Prefix List：掩码的“范围锁定” (ge / le)

在 Prefix List 中，`/len`、`ge` 和 `le` 构成了一个**二维过滤**。

### 核心公式：

假设规则是 `A.B.C.D/len ge X le Y`：

1. **前缀匹配：** IP 的前 `len` 位必须和 `A.B.C.D` 一致。
2. **掩码长度匹配：** 路由条目本身的掩码长度 $M$ 必须满足：$X \le M \le Y$。

### 易混淆点：`ge` 和 `le` 的默认值

很多同学在这里丢分，是因为没记住缺省逻辑：

- **只有 `/len`：** 匹配掩码恰好等于 `len`。
- **只有 `ge X`：** 匹配掩码范围在 `[X, 32]` 之间。
- **只有 `le Y`：** 匹配掩码范围在 `[len, Y]` 之间。

------

## 3. 对比实战：匹配 172.16.0.0 的子网

如果你想匹配 `172.16.0.0` 范围内，掩码在 `24` 到 `26` 位之间的路由：

### ACL 的写法（痛苦且不精准）：

你只能写出 IP 范围，但无法限制掩码长度：

```
access-list 1 permit 172.16.0.0 0.15.255.255
```

- *副作用：* 它会把 `172.16.0.0/16` 这种大网也放进来。

### Prefix List 的写法（优雅且精准）：

```
ip prefix-list TEST permit 172.16.0.0/12 ge 24 le 26
```

- *逻辑：*
  - 第一层：前 12 位必须是 `172.16`（锁定 B 类私有网段）。
  - 第二层：掩码长度必须是 `/24`、`/25` 或 `/26`。

------

## 4. 终极计算陷阱表

| **特性**         | **Wildcard Mask (ACL)** | **Prefix List (ge/le)**      |
| ---------------- | ----------------------- | ---------------------------- |
| **0 的含义**     | **严格匹配**            | N/A (由 `/len` 决定匹配位数) |
| **1 的含义**     | **忽略（任意）**        | N/A                          |
| **掩码长度检查** | **无**（完全不看）      | **核心功能**                 |
| **常用写法**     | `0.0.255.255`           | `/16 ge 24 le 24`            |
| **计算直觉**     | “剩下的位我不管”        | “掩码要在这个范围内”         |

------

### 一个帮你避坑的小技巧

在做 CCNP 题目时，看到 `ge` 和 `le`，你可以直接画一个**数轴**：

- 左端点是 `ge` 的值（如果没有 `ge`，左端点就是 `/len`）。
- 右端点是 `le` 的值（如果没有 `le`，右端点就是 `32`）。
- **只有落在这个区间内的掩码，才是你的 **
