

> **non-client → non-client 会被挡住**

------

# 🔵 场景拓扑

```
        R3
         |
         |
        R2 (RR)
       /   \
     R1     R4
```

假设：

- R2 是 Route Reflector
- R1 是 R2 的 client
- R4 不是 client（non-client）
- R3 也不是 client（non-client）

------

# 🔴 关键场景

假设：

> R3 学到一个新路由 10.10.10.0/24
> 它通过 iBGP 发给 R2

传播路径：

```
R3 → R2(RR) → ?
```

注意：

- R3 是 **non-client**
- R2 是 RR

------

# 🔥 RR 规则触发

RR 从 **non-client** 学到路由时：

```
只发给 client
```

所以：

```
R3 → R2 → R1   ✅
R3 → R2 → R4   ❌
```

R4 收不到。

------

# 🧠 为什么要这样设计？

因为如果允许：

```
non-client → RR → non-client
```

就会出现 **iBGP 环路风险**。

RR 的设计原则是：

> 只允许 client 打破 iBGP 不转发规则。

------

# 🟢 再换一个方向看

如果是：

```
R1 (client) → R2 (RR)
```

那：

```
R1 → R2 → R3   ✅
R1 → R2 → R4   ✅
```

因为：

```
client 进 —— 全部出
```

------

# 🎯 所以你现在明白了吗？

被挡住的情况是：

```
non-client → RR → non-client
```

被允许的情况是：

```
client → RR → anyone
```

------

# 📌 用一句话记住

```
client 进 —— 全放
non-client 进 —— 只给 client
```

------

可以，我按你那道题里**R1 / R2 / R4 / R5**来“照着 RR 规则推演”。（我这里用你题目的描述习惯：**R1 是要把路由对外 eBGP 通告的边界点**，R2、R5 已经被配成 RR，但仍然“路由没送全到 R1”。）

> 你要看的核心就是：**某条路由在某个 RR 上是“从 non-client 学到的”，而 R1 又是 non-client → 就会被挡住。**

------

## 0) 先把 RR 规则写成“判断句”

RR 收到一条 iBGP 路由后：

- 如果这条路由是 **从 client 学到的** → RR 可以发给 **所有 iBGP 邻居**（client + non-client）
- 如果这条路由是 **从 non-client 学到的** → RR **只发给 client**（不会发给其他 non-client）

也就是你熟的那句：

> **non-client 进 → non-client 出：会被挡**

------

## 1) 题目里最关键的“缺路由路径”（按 R5→R2→R1 这条链推）

你可以把内部看成两团：

- **左团**：靠近 R5 / R8 的那一侧（很多内部前缀从这边出来）
- **右团**：靠近 R1（R1 再把这些前缀通告给外部 AS）

题干已经告诉你：**R2 和 R5 是 RR**，但 **R1 还是收不全**，说明一定存在下面这种“坏链路”：

### 假设某个前缀 P 在 R5 那边出现（例如 R8 后面的网段）

传播期望是：

```
P:  R8 → R5(RR) → R2(RR) → R1(边界) → eBGP 出去
```

现在看每一跳会不会被 RR 规则挡住。

------

## 2) 第一处关键：R2 是怎么“学到 P”的？

通常情况下，R2 学到 P 的来源就是 R5（因为 R5 是那一侧的汇聚点）：

```
R2 收到 P：来源 = R5
```

如果 **R5 在 R2 眼里是 non-client**（很常见：RR 之间默认是普通 peer，除非你特意把对方设成 client），那么：

✅ 结论 1：**对 R2 来说，P 是“从 non-client 学到的路由”**。

------

## 3) 第二处关键：R2 会不会把 P 发给 R1？

这一步决定了题目报错“R1 收不全”的根因。

- 若 **R1 在 R2 眼里也是 non-client**
  那 R2 的行为是：

> “我（R2）这条 P 是从 non-client 学到的，所以我只发给我的 clients。”
> R1 不是 client → **不给**。

所以就出现你要的“挡住”：

```
P:  R5(non-client) → R2(RR)  --X-->  R1(non-client)
```

这就是 **non-client → non-client 被挡住** 的最标准形态。

✅ 结论 2：只要同时满足：

- P 在 R2 看来来自 non-client（例如 R5）
- R1 在 R2 看来也是 non-client
  就必然挡住，R1 就缺路由。

------

## 4) 那怎么修？

你会发现修法其实只有两类（本质一样）：

### 修法 A：把 R1 变成 R2 的 client

让 R2 能把“从 non-client 学到的”也发给 R1。

```
R2: neighbor R1 route-reflector-client
```

这样上面那条链就变成：

```
P: R5(non-client) → R2(RR) → R1(client)   ✅能到
```

### 修法 B：让 R1 自己当 RR（题目答案常用写法）

一旦 R1 作为 RR/汇聚点参与反射设计，等价于“确保它能收到全网路由”，题目就喜欢把这个作为正确选项之一。

------

## 5) 那为什么还要 R4（而不是只改 R1）？

因为这类题经常是“双核心两团”：

- 你改了 R1，使它能从 R2 收到更多
- 但另一侧还有一部分路由，可能会走 **R5 ↔ R4 ↔ R1** 或 **R5 ↔ R4 ↔ R2 ↔ R1** 这种链
- 若 R4 在反射关系里位置不对，也会出现同样的 non-client 阻断

最常见的一种“第二处阻断”是：

```
P': R5(non-client) → R4(RR)  --X-->  R1(non-client)
```

所以答案给 **R1 + R4**，本质是在说：

- **R1 必须是“能确保收齐路由”的汇聚点**
- **R4 必须作为另一侧的反射汇聚点**，把 R5 那团的路由可靠地“送到 R1 那团”

------

## 6) 你一句话复述

> 这题错在：R2 从 R5（non-client）学到路由后，按 RR 规则只反射给自己的 clients；如果 R1 不是 client（也是 non-client），R2 就不会把这些路由发给 R1，所以 R1 不全。把 R1 设成 RR/或设成 R2 的 client，并在另一侧用 R4 做汇聚反射，才能保证所有路由最终到 R1 对外通告。
