------

# 🔵 BGP 路由处理流程图（以你那题为例）

假设你在 AS 64511：

```
network 172.16.20.0/24
network 172.16.21.0/24
network 172.16.22.0/24
network 172.16.23.0/24

aggregate-address 172.16.20.0 255.255.252.0 summary-only
```

------

# 🧠 BGP 内部逻辑分 4 步

```
             ┌────────────────────────────┐
             │        Loc-RIB             │
             │  (本地 BGP 路由数据库)       │
             └────────────────────────────┘
                      │
                      ▼
        ┌────────────────────────────┐
        │  1️⃣ Aggregate 生成阶段     │
        │  生成 172.16.20.0/22       │
        └────────────────────────────┘
                      │
                      ▼
        ┌────────────────────────────┐
        │  2️⃣ Suppress 处理阶段      │
        │  summary-only → 抑制 /24    │
        │  suppress-map → 精准抑制     │
        └────────────────────────────┘
                      │
                      ▼
        ┌────────────────────────────┐
        │     Adj-RIB-Out            │
        │  (准备发给 ISP 的路由表)     │
        └────────────────────────────┘
                      │
                      ▼
               发给 ISP 邻居
```

------

# 🔍 每一步到底发生什么？

------

## ① Loc-RIB（本地表）

这里一定会存在：

```
172.16.20.0/24
172.16.21.0/24
172.16.22.0/24
172.16.23.0/24
```

⚠️ 注意：

> summary-only 并不会删除 Loc-RIB 里的 /24
> 它只是控制“对外发布”。

这点很多人误解。

------

## ② Aggregate 生成阶段

BGP 检查：

> 是否存在 172.16.20.0/22 覆盖范围内的 more-specific？

有 → 生成：

```
172.16.20.0/22
```

现在 Loc-RIB 实际上是：

```
/24 × 4
/22 × 1
```

------

## ③ Suppress 阶段（关键！）

### 情况一：没有 summary-only

```
aggregate-address 172.16.20.0 255.255.252.0
```

结果：

```
/24 × 4  ✅ 发
/22 × 1  ✅ 发
```

ISP 会看到 5 条。

------

### 情况二：加 summary-only

```
aggregate-address 172.16.20.0 255.255.252.0 summary-only
```

Suppress 逻辑：

```
/24 × 4  ❌ 不发
/22 × 1  ✅ 发
```

但 Loc-RIB 里仍然有 /24。

------

### 情况三：suppress-map

只 suppress route-map 匹配到的那些。

更精细控制。

------

## ④ Adj-RIB-Out（真正发出去的）

这个阶段才决定：

> ISP 实际看到什么

------

# 🟣 一张更形象的“水流图”

```
/24 routes  →→→→→→→→→→→→→→→→→→→
                     │
                     ▼
                Aggregate
                     │
                     ▼
                生成 /22
                     │
                     ▼
              Suppress 过滤器
                     │
         ┌───────────┴───────────┐
         │                       │
      /24 被挡住             /22 放行
         │                       │
         ▼                       ▼
    不进入 Adj-RIB-Out       进入 Adj-RIB-Out
```

------

# 🎯 你必须记住的核心

### 🔹 BGP 的“发出去”发生在 Adj-RIB-Out

### 🔹 summary-only 只是影响“发”

### 🔹 Loc-RIB 不会被删除

------

# 🧨 再升一级理解（CCIE思维）

真正的顺序是：

```
RIB → BGP table → Aggregate → Best Path → Policy → Adj-RIB-Out
```

aggregate 是在 **BGP table 内部操作**

policy（route-map / filter）是在出接口前操作。

------

# 📌 终极考试口诀

```
aggregate 默认 = 汇总 + 明细一起发
summary-only = 只发汇总
suppress-map = 精准隐藏部分明细
Loc-RIB 不会删，只影响对外发布
```

------

好 👍 这张图是 **BGP 三张表的完整结构图**，配上你刚才的 aggregate 场景一起理解。

------

# 🔵 BGP 三张表完整结构图

```
                ┌───────────────────────────┐
                │        Adj-RIB-In         │
                │  (从邻居收到的所有路由)      │
                └───────────────────────────┘
                              │
                              │  Best Path Selection
                              ▼
                ┌───────────────────────────┐
                │          Loc-RIB          │
                │   (本地最终选中的最佳路径)   │
                └───────────────────────────┘
                              │
                              │  Aggregate / Policy
                              ▼
                ┌───────────────────────────┐
                │        Adj-RIB-Out        │
                │  (准备发给每个邻居的路由)    │
                └───────────────────────────┘
                              │
                              ▼
                         发给 ISP
```

------

# 🔍 每张表在干嘛？

------

## 🟢 1️⃣ Adj-RIB-In

- 存放：从邻居学到的所有路由
- 可能有多条同前缀不同 path
- 还没选最佳

例如：

```
172.16.20.0/24 via ISP-1
172.16.20.0/24 via ISP-2
```

------

## 🟡 2️⃣ Loc-RIB（核心）

- 经过 Best Path 算法
- 每个前缀只留下 **最佳路径**
- 这是“真正用于转发”的路由

例如：

```
172.16.20.0/24 via ISP-1   ← 被选为 best
```

------

## 🔵 3️⃣ Aggregate 在哪里发生？

就在 **Loc-RIB 之后、发出去之前**

假设你本地有：

```
172.16.20.0/24
172.16.21.0/24
172.16.22.0/24
172.16.23.0/24
```

执行：

```
aggregate-address 172.16.20.0 255.255.252.0 summary-only
```

内部会：

```
Loc-RIB:
  /24 × 4
  /22 (新生成)

Suppress 阶段:
  /24 被标记 suppress
```

------

## 🔴 4️⃣ Adj-RIB-Out（真正发出去）

根据 suppress / route-map / prefix-list 过滤后：

```
发送给 ISP:
  172.16.20.0/22
```

/24 仍然存在 Loc-RIB，但不会被发送。

------

# 🧠 一张更“真实”的流程图

```
邻居 → Adj-RIB-In
        │
        ▼
   Best Path 算法
        │
        ▼
     Loc-RIB
        │
        │ aggregate 生成
        ▼
  + 生成 summary
        │
        │ suppress / policy
        ▼
     Adj-RIB-Out
        │
        ▼
     发给邻居
```

------

# 🔥 关键区别总结

| 表          | 作用           | 会被 summary-only 影响吗 |
| ----------- | -------------- | ------------------------ |
| Adj-RIB-In  | 收到的所有路由 | ❌ 不影响                 |
| Loc-RIB     | 最佳路径       | ❌ 不删除                 |
| Adj-RIB-Out | 发给邻居的路由 | ✅ 会过滤                 |

------

# 🎯 考试必杀理解

### BGP 是一个“入 → 选 → 出”的过程

```
入（Adj-RIB-In）
选（Loc-RIB）
出（Adj-RIB-Out）
```

aggregate 影响的是“出”。

------

好，这一题开始进入 **BGP 真正的内部机制层面**。
我给你做一张“结构级对比图”，不是表面命令对比。

------

# 🧠 场景设定

本地有：

```
172.16.20.0/24
172.16.21.0/24
172.16.22.0/24
172.16.23.0/24
```

目标：生成

```
172.16.20.0/22
```

------

# 🔵 一、summary-only 内部机制

配置：

```
aggregate-address 172.16.20.0 255.255.252.0 summary-only
```

------

## 🔬 内部流程图

```
Loc-RIB
  /24
  /24
  /24
  /24
      │
      ▼
Aggregate 生成 /22
      │
      ▼
Suppress 机制启动
      │
      ▼
所有 /24 被标记 suppress
      │
      ▼
Adj-RIB-Out 只剩 /22
```

------

## 🔎 关键点

- 所有 more-specific 全部 suppress
- 不做匹配
- 不看 policy
- 是一个“全局 blanket 抑制”

### 你可以理解为：

> summary-only = 自动隐藏所有明细

------

# 🔴 二、suppress-map 内部机制

配置：

```
aggregate-address 172.16.20.0 255.255.252.0 suppress-map BLOCK
```

------

## 🔬 内部流程图

```
Loc-RIB
  /24
  /24
  /24
  /24
      │
      ▼
Aggregate 生成 /22
      │
      ▼
检查 route-map BLOCK
      │
      ├── 匹配到 → suppress
      └── 没匹配 → 正常发
      ▼
Adj-RIB-Out
```

------

## 🔎 关键点

- 只 suppress 被 route-map 匹配到的
- 没匹配到的明细仍然发出去
- 可以做到“部分隐藏”

------

# 🟣 一张真正的对比结构图

```
                 ┌──────────────┐
                 │   Loc-RIB    │
                 └──────────────┘
                          │
                          ▼
                    生成 summary
                          │
         ┌────────────────┴────────────────┐
         │                                 │
   summary-only                       suppress-map
         │                                 │
  所有明细自动 suppress            route-map 决定 suppress 谁
         │                                 │
         ▼                                 ▼
   Adj-RIB-Out 只发 summary      Adj-RIB-Out 发 summary + 未匹配明细
```

------

# 🔥 举个真实工程场景

### summary-only 用在：

- ISP 对外发布企业前缀
- 企业边界只允许发汇总
- 简化路由表

------

### suppress-map 用在：

- 你想隐藏大部分 /24
- 但某几个 /24 必须单独对外发布
- 做流量工程
- 做特殊 prefix 公告

例如：

```
只 suppress 172.16.21.0/24
保留 172.16.22.0/24 对外发布
```

这是 summary-only 做不到的。

------

# 🎯 最核心区别一句话

```
summary-only = 全部隐藏
suppress-map = 精准隐藏
```

------

# 🧨 再升一级（很多人不知道）

summary-only 实际内部也是：

> 自动创建一个隐式 suppress-map（匹配所有 more-specific）

只是你看不到。

------

# 📌 考试记忆模型

| 维度           | summary-only | suppress-map   |
| -------------- | ------------ | -------------- |
| 控制精度       | 全部         | 可选择         |
| 需要 route-map | ❌            | ✅              |
| 典型用途       | ISP 汇总     | 精准流量工程   |
| 风险           | 简单安全     | 配错会泄露明细 |

------
