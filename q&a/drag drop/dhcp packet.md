| **DHCP 报文类型** | **描述 (Description)**                                       | **方向**   | **核心功能**                                       |
| ----------------- | ------------------------------------------------------------ | ---------- | -------------------------------------------------- |
| **DHCPACK**       | **server-to-client communication with configuration parameters...** | **S -> C** | 最终确认，发送 IP 和配置参数。                     |
| **DHCPINFORM**    | **client-to-client communication, asking for only local configuration parameters...** | **C -> S** | 客户端已有 IP，只想请求 DNS/网关等额外参数。       |
| **DHCPNAK**       | **server-to-client communication, refusing the request...**  | **S -> C** | 拒绝请求（例如 IP 已租给别人或客户端移动了子网）。 |
| **DHCPDECLINE**   | **client-to-server communication, indicating that the network address is already in use** | **C -> S** | 客户端检测到 IP 冲突（通过 ARP），拒绝使用该 IP。  |

你可以通过词根和“谁发给谁”来快速判断：

- **ACK vs NAK (服务器发的判决书)**：
  - **ACK** (Acknowledgment)：肯定答复，给你 IP。
  - **NAK** (Negative Acknowledgment)：否定答复，拒绝给你这个 IP。
- **INFORM (客户端在咨询)**：
  - 客户端已经通过其他方式（比如静态配置）有了 IP，它只是想 **Inform** (告知/咨询) 服务器：*“嘿，我 IP 有了，能告诉我 DNS 是多少吗？”*
- **DECLINE (客户端在婉拒)**：
  - 服务器给了个 IP，但客户端发现局域网里有人在用（ARP 冲突），于是 **Decline** (谢绝) 了这个 Offer。

**DHCPDECLINE 的触发条件**：客户端在收到 DHCPACK 后，通常会发送 **Gratuitous ARP (无故 ARP)**。如果收到回应，说明地址冲突，此时才会发送 DHCPDECLINE。

**DHCPINFORM 的应用场景**：最常见于客户端已经有静态 IP 地址，但需要通过 DHCP 获取 WPAD（代理自动发现）或选项 150（TFTP 服务器地址）等配置。

| **报文名称**    | **方向**  | **核心作用**                                                 |
| --------------- | --------- | ------------------------------------------------------------ |
| **DHCPNAK**     | **S → C** | **拒绝**：服务器告诉客户端“这个 IP 你不能用”（例如租约过期或客户端换了子网）。 |
| **DHCPDECLINE** | **C → S** | **婉拒**：客户端通过 ARP 发现 IP 已被占用，主动拒绝服务器给的这个 IP。 |
| **DHCPRELEASE** | **C → S** | **释放**：客户端正常下线，主动告诉服务器回收 IP 地址。       |

| ---------------- | --------- | ------------------------------------------------------------ |
| **DHCPDISCOVER** | **C → S** | **寻找**：客户端在子网内广播，寻找可用的 DHCP 服务器。       |
| **DHCPOFFER**    | **S → C** | **响应**：服务器预留一个 IP，并向客户端发送包含 IP、掩码、租期等信息的提议。 |
| **DHCPREQUEST**  | **C → S** | **请求**：客户端选择一个 Offer（通常是第一个达到的），并广播告知所有服务器它的选择。 |

**为什么 REQUEST 是广播？**

- 在 DORA 流程中，客户端发送 `DHCPREQUEST` 是为了通知**所有**潜在的服务器：“我已经选了 A 服务器的 IP，B 和 C 你们可以把预留的 IP 收回去了。”

**DHCPDECLINE 的触发点**：

- 客户端在收到 `DHCPACK` 后会执行 **Gratuitous ARP (无故 ARP)**。如果有人回应，说明冲突，此时才会发出 `DHCPDECLINE`。

**续租 (Renewal) 的逻辑**：

- 当租期到达 **50% (T1)** 时，客户端会发送单播 `DHCPREQUEST` 给原服务器。
- 如果没回应，到达 **87.5% (T2)** 时，会发送广播 `DHCPREQUEST` 寻找任何可用的服务器。
