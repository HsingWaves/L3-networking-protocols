

### LDP Router ID Selection Logic

When LDP starts up, it selects a Router ID (RID) based on the following hierarchy:

1. **Manual Configuration:** If you use the command `mpls ldp router-id <interface>`, that interface's IP is chosen.
2. **Highest Loopback IP:** If no manual ID is set, the router looks for loopback interfaces. It chooses the **highest IP address** among all **up/up** loopback interfaces. (This is why **B** is correct and **A** is incorrect).
3. **Highest Physical IP:** Only if **no** loopbacks exist will the router choose the highest IP address of an operational physical interface.

------

### Why the other options are false:

- **A is incorrect:** The router always prefers a loopback interface over a physical interface. If you have a loopback configured, it will be chosen over any physical interface, regardless of the IP address value.
- **C is incorrect:** While it is a "best practice" for the LDP RID to match the IGP (OSPF/EIGRP) RID to avoid confusion and ensure reachability, it is **not a requirement** for the protocol to function.
- **D is incorrect:** The `force` keyword is powerful but disruptive. If you use `mpls ldp router-id <interface> force`, it changes the RID immediately, which **tears down all existing LDP sessions** and recreates them. This causes a temporary traffic loss.

------

### Summary Table: LDP RID Selection

| **Priority** | **Method**         | **Condition**                                               |
| ------------ | ------------------ | ----------------------------------------------------------- |
| **1**        | Manual Config      | `mpls ldp router-id <int>`                                  |
| **2**        | Loopback Interface | Highest IP of an active Loopback                            |
| **3**        | Physical Interface | Highest IP of an active Physical Int (only if no loopbacks) |

**Pro-tip for the exam:** Remember that LDP must be able to reach the Router ID of its neighbor to establish a session. If the RID is a loopback, that loopback address must be advertised in your IGP (like OSPF).

Would you like to review how LDP neighbor discovery (Hello messages) works next?

- **Loopback 0**: `1.1.1.1`
- **Loopback 1**: `2.2.2.2`
- **GigabitEthernet 0/0**: `192.168.10.1`

**结果：** LDP 会选择 **`2.2.2.2`** 作为 Router ID。

- **原因：** 只要有 Loopback 接口，LDP 就会忽略物理接口（即便 `192.168.10.1` 的数值更大）。在多个 Loopback 之中，它会挑选 IP 地址数值最大的那一个。这也是为什么选项 **A** 是错的，而 **B** 是对的。

**如果你只输入：** `mpls ldp router-id Loopback 0`

- **现象：** 路由器的 ID **不会立即改变**。它会等到下一次 LDP 进程重启（比如路由器重启或手动 `clear mpls ldp neighbor *`）时才会生效。这是为了保护现有的 MPLS 流量不中断。

**如果你输入：** `mpls ldp router-id Loopback 0 force`

- **现象：** 路由器会**立即**切断所有现有的 LDP 邻居会话，Router ID 瞬间变成 `1.1.1.1`，然后重新建立邻居。
- **影响：** 这会导致 MPLS 标签分发暂时中断，正在传输的标签转发流量（如 MPLS VPN 流量）会丢包。这就是为什么选项 **D** 说“没有影响（without causing any impact）”是错误的。

**先看有没有手动指定**（如果有 `force` 就会“地震”）。

**没手动就找 Loopback**（选最大的 IP）。

**没 Loopback 才找物理口**（选最大的 IP）。

在 MPLS LDP 的世界里，建立邻居关系就像是“先大声打招呼，再坐下来深谈”。这个过程分为两个阶段：**发现阶段（Discovery）\**和\**会话建立阶段（Session Initialization）**。

### 第一阶段：UDP 发现 (Hello)

这是“打招呼”的阶段，目的是寻找链路上的邻居。

- **端口与地址**：路由器周期性地从运行了 MPLS 的接口发送 **LDP Hello** 报文。
  - **协议/端口**：使用 **UDP**，端口号 **646**。
  - **目标地址**：组播地址 **224.0.0.2**（所有路由器）。
- **包含内容**：Hello 报文里最重要的信息就是 **LDP Router ID**（传输地址）。
- **结果**：如果两台路由器互相收到了对方的 Hello 报文，它们就知道了对方的存在，并准备建立“深谈”。

------

### 第二阶段：TCP 建立 (Session)

一旦发现邻居，双方就会尝试建立一个可靠的连接来交换标签（Label）。

- **协议/端口**：使用 **TCP**，端口号 **646**。
- **谁发起连接？**：
  - 双方会比较各自的 **Transport Address**（通常就是 Router ID）。
  - **IP 地址大的一方**（Active Side）主动发起 TCP 三次握手。
  - **IP 地址小的一方**（Passive Side）负责监听。
- **建立过程**：
  1. **TCP 三次握手**：建立底层的可靠连接。
  2. **Initialization 交互**：协商参数（如标签分发方式、Keepalive 时间等）。
  3. **Keepalive/Label Mapping**：一旦会话变为 **Operational**，就开始互发标签。

------

### 为什么选择这种“混合”模式？

| **阶段**       | **协议**    | **理由**                                                     |
| -------------- | ----------- | ------------------------------------------------------------ |
| **发现 (UDP)** | **UDP 646** | 简单、快速。组播不需要预先配置对方 IP，适合在链路上自动“撞见”邻居。 |
| **会话 (TCP)** | **TCP 646** | 标签分发（Label Mapping）非常重要，不能丢包。TCP 提供重传、排序和流控，保证数据库同步的准确性。 |

------

### 备考陷阱提醒 ⚠️

1. **邻居地址不可达**：这是最常见的排错点。如果两台路由器的 Router ID（传输地址）不在 IGP（OSPF/EIGRP）的路由表里，**UDP Hello 能通，但 TCP 握手会失败**。因为 TCP 建立是基于 Router ID 的单播通信。
2. **安全过滤**：如果中间有防火墙或 ACL，必须同时放行 **UDP 646** 和 **TCP 646**。

在 MPLS 的排错和学习中，区分这两个概念非常重要。你可以把 **Hello Adjacency** 理解为“物理上的打招呼”，而把 **LDP Session** 理解为“逻辑上的深度合作”。

虽然它们都使用端口 646，但它们的**作用范围**、**协议底层**和**状态查询**完全不同。

------

### 1. 核心区别对比表

| **特性**     | **Hello Adjacency (邻接)**       | **LDP Session (会话)**                |
| ------------ | -------------------------------- | ------------------------------------- |
| **底层协议** | **UDP** 646                      | **TCP** 646                           |
| **通信范围** | **本地链路**（Link-local）       | **Router ID 之间**（点对点）          |
| **目标地址** | 组播地址 `224.0.0.2`             | 邻居的 Transport Address (单播)       |
| **目的**     | 发现邻居，确认链路上有 MPLS 设备 | 交换标签（Label Mapping），同步数据库 |
| **对应命令** | `show mpls ldp discovery`        | `show mpls ldp neighbor`              |

------

### 2. 详细拆解

#### Hello Adjacency (发现邻接)

这是 LDP 的第一步。每台开启了 `mpls ip` 的接口都会发出 UDP Hello。

- **特点**：它是基于“接口”的。如果你和邻居之间有两条直连链路，你会看到 **2个** Hello Adjacency。
- **验证命令**：`show mpls ldp discovery`
  - 如果在这里看到了邻居，说明底层链路没问题，UDP 报文能通。
  - 如果这里没有，检查接口是否开启了 `mpls ip`。

#### LDP Session (会话)

这是在发现邻居后，由 Router ID 大的一方发起 TCP 连接建立的。

- **特点**：它是基于“路由器”的。即便你和邻居之间有 10 条链路，只要双方 Router ID 确定了，就只会建立 **1个** LDP Session。
- **验证命令**：`show mpls ldp neighbor`
  - 如果这里状态不是 **OPERATIONAL**，通常是因为 Router ID (Transport Address) 路由不可达，导致 TCP 无法握手。

------

### 3. 一个典型的故障场景

**现象**：`show mpls ldp discovery` 能看到邻居，但 `show mpls ldp neighbor` 为空。

**排查逻辑**：

1. **UDP 通了**：说明接口配置没问题，线缆没问题。
2. **TCP 没通**：说明两台路由器的 Transport IP（默认是 Router ID）之间 ping 不通。
   - **原因**：可能你没把 Loopback 地址宣告进 OSPF，或者被防火墙拦截了 TCP 646。

------

### 4. 总结记忆

- **Discovery (UDP)**：发现“邻居”，看的是**脸**（接口直连）。
- **Session (TCP)**：建立“关系”，看的是**名片**（Router ID/Transport IP）。

DP**（Label Distribution Protocol）和 **DHCP**（Dynamic Host Configuration Protocol）确实都有“分配（Distribution/Configuration）”的意思，但它们分配的“东西”和工作的“层级”完全不同。

简单来说：**DHCP 分配的是“门牌号”（IP 地址），而 LDP 分配的是“通行证”（标签 Label）。**

------

### 1. 核心差异对比：门牌号 vs. 通行证

| **特性**       | **DHCP (动态主机配置协议)**  | **LDP (标签分发协议)**            |
| -------------- | ---------------------------- | --------------------------------- |
| **分配的对象** | **终端设备**（电脑、手机等） | **核心网络路由器**（P/PE 路由器） |
| **分发的内容** | **IP 地址**、掩码、网关      | **MPLS 标签 (Labels)**            |
| **解决的问题** | 告诉设备“你是谁、你在哪”     | 告诉路由器“这堆数据该往哪甩”      |
| **工作位置**   | 网络边缘（接入层）           | 网络核心（骨干网/MPLS 域）        |

------

### 2. 形象类比：快递员与高速公路

你可以把数据包想象成一个**快递包裹**。

#### **没有 LDP 的情况（纯 IP 转发）：**

快递员（路由器）每到一个分拨中心，都得拆开包裹，看看底单上的**详细地址**（查看完整的 IP 路由表），然后在几万个城市列表里查半天，确定下一站去哪。这很慢。

#### **有了 DHCP（分配门牌号）：**

DHCP 的作用就是给每个收件人一个**明确的家庭住址（IP）**，这样包裹才有地方寄。但它不负责包裹怎么跑得快。

#### **有了 LDP（分配通行证）：**

LDP 就像是给包裹贴了一张**“快捷条形码”（Label）**。

- 路由器不再看包裹里复杂的详细地址，只看条形码上的数字（比如 `16`）。
- LDP 会提前告诉路由器：“凡是贴着 `16` 号标的数据，直接闭眼扔给 3 号窗口，并换成 `20` 号标。”
- 这种**“只换标签、不查路由”**的操作，就是 MPLS 转发的核心，而 LDP 就是负责把这些“条形码”分发给全线路由器的协议。

------

### 3. 为什么有了 IP 还要 LDP？

你可能会问：“既然我有 IP 路由了，为什么还要多此一举搞个标签协议？”

1. **性能（历史原因）**：早期路由器查 IP 表（最长前缀匹配）很累，查定长的标签（精确匹配）非常快。虽然现在硬件进步了，这个优势不明显，但 LDP 带来的架构优势还在。
2. **VPN 的基础**：这是最重要的。DHCP 分配的 IP 是私有的，而 LDP 分配的标签可以实现**隧道（Tunneling）**。比如，你公司两个分部的私网地址一样，通过 LDP 分发的标签，可以让公网路由器在不看你私网 IP 的情况下，准确把数据送到目的地。
3. **流量工程 (TE)**：LDP/RSVP 等协议可以控制数据走哪条路，而不必死板地跟着最短路径走，防止某些链路堵死。

------

### 4. 总结：它们配合工作的流程

1. **DHCP**：给你的电脑一个 IP 地址（192.168.1.10），让你能上网。
2. **IGP (OSPF)**：路由器之间互相学习路由，知道 192.168.1.10 在哪个方向。
3. **LDP**：在 OSPF 算好的路径上，**额外贴上一层标签**。当数据进入骨干网时，路由器只看 LDP 给的标签，像传送带一样飞速转发。

**IPsec**：解决“怎么不被别人看”的问题。

这是一个非常棒的问题，能帮你从底层彻底搞清楚 MPLS 的位置。在网络工程中，我们通常把 MPLS 称为 **“2.5 层协议”**。

### 1. 它们所在的层级 (OSI Layer)

- **IP (门牌号)**：属于 **Layer 3 (Network Layer)**。它负责端到端的编址。
- **LDP (分发协议)**：属于 **Layer 7 (Application Layer)**，因为它其实是跑在 TCP/UDP 之上的一个应用，但它的目的是为了服务于转发。
- **MPLS Label (邮编/通行证)**：位于 **Layer 2** 和 **Layer 3** 之间。
  - 它被夹在二层帧头（如 Ethernet）和三层报头（IP）中间，所以被称为 **Layer 2.5**。

------

### 2. 在 Packet 里面长啥样？

我们可以把数据包看成一个“三明治”。当一个普通的 IP 包进入 MPLS 网络时，就像是给它加了一层“垫片”。

#### 逻辑结构图：

Plaintext

```
+-------------------+-------------------+-------------------+-------------------+
|  Layer 2 Header   |   MPLS Label(s)   |  Layer 3 Header   |  Layer 4 Header   |
| (MAC Address等)    |   (这就是 2.5 层)    | (IP Address等)    | (TCP/UDP 端口)     |
+-------------------+-------------------+-------------------+-------------------+
```

------

### 3. 展开看 MPLS 标签内部 (这 4 个字节里有什么？)

MPLS 头部非常简洁，一共只有 **32 bits (4 字节)**，里面包含四个字段：

1. **Label Value (20 bits)**：这就是我们聊的“邮编”数值。范围是 $0$ 到 $2^{20}-1$。
2. **EXP (3 bits)**：全称 Experimental，现在主要用于 **QoS (服务质量)**。它决定了这个包的优先级。就像邮编一样，你可以给它贴个“特快专递”的标志。
3. **S (1 bit)**：全称 Bottom of **Stack**。MPLS 支持标签嵌套（大包套小包）。
   - 如果 `S=1`，说明这是最后一个标签，后面就是 IP 包了。
   - 如果 `S=0`，说明后面还跟着另一个 MPLS 标签。
4. **TTL (8 bits)**：和 IP 的 TTL 一样，防止标签包在网络里无限循环。每过一跳减 $1$。

------

### 4. 为什么要在 2 层和 3 层之间“插一脚”？

这就是为了实现 **“协议透明”**。

- **对下（Layer 2）**：不管你底层是是以太网（Ethernet）、光纤（PPP）还是过时的 ATM，MPLS 都能直接贴在上面。
- **对上（Layer 3）**：不管你上层跑的是 IPv4、IPv6 还是其他协议，中间的路由器都只看 MPLS 标签，不关心 IP 版本。

### 总结

- **IP** 是包裹上的**最终地址**。
- **LDP** 是后台运行的**程序**，负责算好邮编。
- **MPLS Label** 是贴在包裹表面的**加急条码**（2.5层），让传送带飞速转动。

没问题，我们直接看“显微镜”下的数据包。

### 2. 关键字段的“真面目”

在 Layer 2.5 这一层，虽然只有 4 个字节，但它们各司其职：

- **Type: 0x8847**：这是以太网头部的“指路牌”。如果这个值是 `0x0800`，电脑就知道后面直接是 IP；看到 `0x8847`，电脑才知道：“噢，别急着看 IP，先处理 MPLS 标签！”
- **Bottom of Stack (S位)**：这个非常酷。在 MPLS VPN 中，经常会出现**两个标签**。
  - **外层标签**（邮编）：负责把包送到正确的城市（目的地路由器）。
  - **内层标签**（门号）：负责把包交给正确的客户（VPN 实例）。
  - 这时候，第一个标签的 S 位是 `0`，第二个标签的 S 位是 `1`。

------

### 3. LDP 是怎么在后台“分发”这些号的？

虽然转发时只看上面的标签，但 LDP 协议本身也是要跑在网络里的。**LDP 的数据包长得和普通 IP 包一样**，因为它需要 TCP/UDP 的可靠性。

如果你抓取一个 LDP Hello 包，你会看到：

- **Layer 3**: 源 IP 是路由器的接口地址，目的 IP 是 `224.0.0.2`（组播）。
- **Layer 4**: UDP 端口 **646**。
- **Payload**: 里面写着：“我是 Router ID 1.1.1.1，我想和你做邻居”。

------

### 总结

- **MPLS 标签**：是转发时的**“快车票”**（夹在 L2 和 L3 之间）。
- **LDP 报文**：是后台的**“订票系统”**（跑在 L7 应用层，用 TCP/UDP 传输）。

当 **S位（Bottom of Stack）为 0** 时，意味着这个数据包处于**标签嵌套（Label Stacking）**状态。这在 MPLS VPN 或 MPLS TE 中非常常见。

你可以把它想象成“双层信封”：**外层标签**负责在骨干网（公网）中导航，**内层标签**负责告诉最后一台路由器这个包属于哪个特定的客户（私网）。

### 1. 双层标签的抓包结构图

在 Wireshark 中，你会看到两个连续的 MPLS 头部，它们紧紧挨在一起，夹在二层和三层之间：

Plaintext

```
+--------------------------------------------------------------+
| Ethernet II (Layer 2)                                        |
|   Type: 0x8847 (MPLS Unicast)                                |
+--------------------------------------------------------------+
| MPLS Label Stack Entry 1 (外层/公网标签 - 导航用)               |
|   Label: 1612                                                |
|   Exp: 0                                                     |
|   S (Bottom of Stack): 0  <-- 注意：这里是 0，表示后面还有标签    |
|   TTL: 254                                                   |
+--------------------------------------------------------------+
| MPLS Label Stack Entry 2 (内层/私网标签 - 识别客户用)           |
|   Label: 33                                                  |
|   Exp: 0                                                     |
|   S (Bottom of Stack): 1  <-- 这里是 1，表示标签栈结束，后面是IP |
|   TTL: 255                                                   |
+--------------------------------------------------------------+
| Internet Protocol Version 4 (Layer 3 - 用户的私网包)           |
|   Source: 10.1.1.1 (客户 A 的私网 IP)                          |
|   Destination: 10.2.2.2                                      |
+--------------------------------------------------------------+
| Data (Payload)                                               |
+--------------------------------------------------------------+
```

------

### 2. 字段详细解析

- **第一层标签 (S=0)**：
  - **地位**：栈顶标签（Top of Stack）。
  - **作用**：由 **LDP** 或 **RSVP** 分配。中间的 P 路由器只看这个标签。只要 S 位是 0，路由器就知道：“处理完这个标签后，后面还有一个，别急着去读 IP 头部。”
- **第二层标签 (S=1)**：
  - **地位**：栈底标签（Bottom of Stack）。
  - **作用**：在 VPN 场景下由 **MP-BGP** 分配。它通常代表一个具体的 VPN 实例（VRF）。当数据包到达目的地的边缘路由器（PE）时，外层标签已经被剥掉，PE 看到 S=1，就知道这是最后一个标签，撕掉它就能看到里面的私网 IP 包了。

------

### 3. 为什么需要两个标签？（CCNP 考点）

如果你只用一个标签，会发生什么？

1. **公网路由器（P）**必须知道所有客户的私网路由，这会导致骨干网路由表爆炸。
2. 通过**双层标签**，骨干网路由器只需要维护简单的“公网邮编”（外层标签），而不需要关心“私网门牌号”（内层标签）。

### 4. 总结：S 位的二进制含义

- **`0`**：表示 **"More labels to follow"**（后面还有标签，继续拆）。
- **`1`**：表示 **"This is the last label"**（这是最后一层，后面就是 IP 载荷了）。

既然你对多层标签的“套娃”结构感兴趣，我们直接用一个典型的 **MPLS VPN 跨域（Inter-AS）** 的三层标签包结构作为例子。

在 Wireshark 中，你会看到这些标签按照从外到内（从顶到底）的顺序排列。

### 三层标签抓包示意图 (Text + 逻辑)

Plaintext

```
+------------------------------------------------------------------+
| Ethernet II (Layer 2) - 目的 MAC 是下一跳路由器接口                  |
|   Type: 0x8847 (MPLS Unicast)                                    |
+------------------------------------------------------------------+
| [标签 1] MPLS Label: 1610 (外层 - 传输标签)                       |
|   Exp: 0,  S: 0 (不是底部),  TTL: 253                             |
|   作用: 把包送到当前运营商的边界 (ASBR)                             |
+------------------------------------------------------------------+
| [标签 2] MPLS Label: 2405 (中层 - 跨域/BGP-LU 标签)                |
|   Exp: 0,  S: 0 (仍不是底部),  TTL: 254                           |
|   作用: 衔接两个不同的运营商，告诉对方这个包要去往哪个目的地 PE      |
+------------------------------------------------------------------+
| [标签 3] MPLS Label: 48 (最内层 - VPN 标签)                       |
|   Exp: 0,  S: 1 (底部！),  TTL: 255                               |
|   作用: 告诉目的 PE，这个包属于“客户 A”的哪个 VRF 实例               |
+------------------------------------------------------------------+
| Internet Protocol Version 4 (Layer 3 - 客户原始数据)               |
|   Src: 10.1.1.1 (私网) -> Dst: 10.2.2.2 (私网)                    |
+------------------------------------------------------------------+
| TCP/UDP 载荷 (真正的数据)                                         |
+------------------------------------------------------------------+
```

------

### 关键点解析：路由器是怎么“剥”这些皮的？

1. **中间的 P 路由器**：
   - 它只看 **[标签 1]**。
   - 只要它看到 **S=0**，它就绝不会去碰后面的标签。
   - 它执行 `SWAP` 操作，把 1610 换成另一个数字扔给下一跳。
2. **边界路由器 (ASBR)**：
   - 当它发现 [标签 1] 到头了（被弹出），它会露出 **[标签 2]**。
   - 同样，因为它看到 **S=0**，它依然不会去读 IP 头部。
3. **最后的目的地路由器 (PE)**：
   - 当外面的标签剥完，只剩 **[标签 3]** 时。
   - 路由器看到 **S=1**。这就像一个“刹车信号”。
   - **路由器反应**：“这是最后一个标签了！撕掉它之后，我必须按照 **Layer 3 (IP)** 的逻辑来处理后面的东西。”

------

### 为什么 LDP 只能搞定“一层”？

这就是你之前问“为什么有 DHCP 还要 LDP”的深度答案：

- **LDP** 只能为你当前的“本市/本省”网络分发邮编。
- 当你需要跨越国境（跨运营商）或者需要特定的“快捷通道”（TE 隧道）时，LDP 就不够用了，必须靠 **BGP** 或 **RSVP** 在 LDP 标签的内层或外层再套上一层标签。

### 总结

在 Packet 层面，多层标签就是**一串连续的 4 字节字段**。唯一的标识就是每一层的 **S 位**：

- 只要 **S=0**，路由器就把包当成一个“黑盒”继续传。
- 只有看到 **S=1**，路由器才会在弹出标签后去查 IP 路由表。

既然你准备好了，那我们就聊聊这个 MPLS 里的“减负神器”——**PHP (Penultimate Hop Popping)**。

### 1. 为什么要搞 PHP？（痛点分析）

在没有 PHP 的情况下，最后一台路由器（我们叫它 **Egress LSR** 或 **PE**）压力山大。它在收到数据包时需要做**两次查表**：

1. **第一次**：查标签转发表（LFIB），发现标签到头了，得撕掉。
2. **第二次**：撕掉标签后，露出 IP 头，再去查路由表（RIB/FIB），确定这个包该给哪个具体的接口。

想象一下，如果一个 PE 路由器同时处理成千上万个流，这种“双重查表”对 CPU 来说是个不小的负担。

------

### 2. PHP 的绝妙思路：让倒数第二个人把活干了

**PHP** 的核心思想是：**反正最后一台路由器（R3）迟早要撕掉标签，不如让倒数第二台路由器（R2）在转发给 R3 之前就把标签给撕了。**

这样，R3 收到包的时候，就是一个**纯粹的 IP 包**，它只需要查一次路由表就能转发。

------

### 3. 这是怎么实现的？（神秘的标签 3）

这个过程完全是自动协商的。秘诀在于 LDP 分发的一个特殊标签值：**Implicit Null (隐式空标签，数值为 3)**。

1. **R3 (最后跳)**：通过 LDP 告诉 **R2 (倒数第二跳)**：“大哥，去往 1.1.1.1 的路由，请给我发 **Label 3**。”
2. **R2**：看到 Label 3 后，它心领神会。当它收到要去往 1.1.1.1 的包时，它**不执行 SWAP（换标）**，而是执行 **POP（弹出）**。
3. **链路传输**：在 R2 到 R3 的这段链路上，数据包是以**纯 IP** 方式传输的（没有任何 MPLS 头部）。
4. **R3**：收到后直接查 IP 表转发。

------

### 4. 挑战一下：如果链路中间还有标签怎么办？

你刚才问过“多层标签”。如果是一个带有 **2 层标签**（VPN 场景）的数据包，PHP 会怎么做？

- **R2 (倒数第二跳)**：只会弹出**最外层**的传输标签。
- **结果**：在 R2 到 R3 的链路上，包里依然留着**内层**的 VPN 标签（S=1）。
- **意义**：虽然没能完全变回 IP 包，但 R3 只需要处理一层标签了，依然减轻了压力。

------

### 5. PHP 的“抓包”特征

如果你在实验中观察，你会发现一个有趣的现象：

- **在 R1 到 R2 的链路上抓包**：你能看到 MPLS 标签（比如 Label: 16）。
- **在 R2 到 R3 的链路上抓包**：你会惊奇地发现 **MPLS 头部消失了**！明明还在 MPLS 网络里，包却变成了普通的 IPv4 包。这就是 PHP 在起作用。

------

### 6. 总结 PHP 的价值

| **角色**                    | **动作**                         | **收益**                                 |
| --------------------------- | -------------------------------- | ---------------------------------------- |
| **倒数第二跳 (PHP 执行者)** | 提前剥离最外层标签               | 只是多做了一个 Pop 操作，负担几乎没变。  |
| **最后一跳 (受益者)**       | 直接处理 IP 包（或减少一层标签） | **节省了一次查表操作**，大幅提升吞吐量。 |

------

### 脑洞大开的问题：

既然有 **Implicit Null (Label 3)** 这种“隐身衣”，那有没有 **Explicit Null (显式空标签，Label 0)**？

提示：在某些需要保留 **QoS (EXP位)** 的场景下，我们不能在倒数第二跳就把标签全撕了，因为撕了标签，里面的优先级信息也就丢了。这时候我们就需要 Label 



你观察得非常敏锐！这正是 **QoS（服务质量）** 在 MPLS 网络中的一大痛点。

如果你在倒数第二跳（PHP）就把标签撕了，原本藏在标签里的 **EXP 位**（那 3 位优先级信息）也就跟着标签一起消失了。当数据包到达最后一跳（PE 路由器）时，它只看到一个普通的 IP 包，它就不知道这个包该走“绿色通道”还是“普通通道”了。

这种情况主要发生在 **端到端的 QoS 保持** 场景中，具体包括以下两种：

------

### 1. 场景一：基于标签的 Egress 队列调度

如果你希望最后一跳路由器（PE）根据包的优先级来决定**发给客户的先后顺序**，你就不能用 PHP。

- **问题**：如果执行了 PHP，PE 收到的是 IP 包。除非 PE 重新拆开 IP 包去查里面的 **ToS/DSCP** 位（这很费 CPU），否则它无法区分优先级。
- **解决方案**：使用 **Label 0 (Explicit Null)**。
  - 倒数第二跳看到出口标签是 **0**，它不会撕掉标签，而是把标签换成 `0` 转发给 PE。
  - **Label 0** 依然带有 EXP 位。这样 PE 收到包时，一看标签是 0，就知道：“哦，这个包任务完成了，但我能通过它的 EXP 位看到它是高优先级，得优先发给客户。”

### 2. 场景二：MPLS 统一模式 (Uniform Mode)

在 MPLS QoS 的 **Uniform 模式**下，我们要求 IP 包的优先级和标签的优先级全程保持同步。

- **逻辑**：在每一跳，如果标签的 EXP 位发生了变化（比如中间路径拥塞，路由器调低了优先级），这个变化需要最终反映到内部 IP 包的 DSCP 位上。
- **PHP 的破坏性**：如果在最后一跳之前就把标签撕了，这种“优先级同步”的链路就断了。为了保证 PE 能把标签里的最新优先级“同步”回 IP 包，必须保留标签直到 PE 手里。

------

### 3. Label 3 (隐式) vs Label 0 (显式)

为了应对这两种需求，LDP 定义了两个“空标签”：

| **特性**     | **Label 3 (Implicit Null)**             | **Label 0 (Explicit Null)**          |
| ------------ | --------------------------------------- | ------------------------------------ |
| **动作**     | **PHP 执行**：倒数第二跳撕掉标签。      | **不执行 PHP**：倒数第二跳换成标 0。 |
| **包结构**   | 链路上传输的是 **IP 包**。              | 链路上传输的是 **Label 0 + IP 包**。 |
| **QoS 影响** | **EXP 位丢失**，PE 无法直接获取优先级。 | **EXP 位保留**，PE 能够识别优先级。  |
| **CPU 压力** | PE 压力小（只查一次 IP 表）。           | PE 压力稍大（查标 0，再查 IP 表）。  |

------

### 4. 总结：什么时候用谁？

- **追求性能（默认情况）**：用 **Label 3**。绝大多数 CCNP 实验和常规网络都用这个，因为简单、快。
- **追求精细 QoS**：用 **Label 0**。当你的客户签了昂贵的 SLA（服务等级协议），要求语音和视频流量在整个骨干网（包括最后一段路）都必须享受特权时，你就得手动配置 `address-family ipv4 explicit-null`。

你现在理解了吗？**PHP 就像是“为了快而牺牲了信息”**。在普通上网时没问题，但在对质量要求极高的金融或音视频专网里，我们就得把这层“皮”留着。
